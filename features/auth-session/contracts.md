# Contracts: Auth Session Refactor

> All HTTP endpoints, headers, and DTOs that touch the consolidated auth-session world. Source of truth for the wire format — frontend models and Flutter port must match exactly.

---

## 1. The single boot endpoint

### `GET /session`

Returns the consolidated [`Session`](../../../packages/session/src/lib/session.dto.ts) snapshot used by `SessionStore`. Replaces the legacy triple `GET /auth/current` + `GET /auth/context` + `GET /tenants/current/features` plus the per-tenant `GET /tenants/current/push-brand` + `GET /tenants/current/theme` and the per-employee `GET /employees/me/preferences` round-trip.

**Auth:** Bearer JWT. Backend Guard: standard `AuthGuard` (any authenticated principal).

**Request headers:**

| Header | Required | Purpose |
| --- | --- | --- |
| `Authorization: Bearer <token>` | yes | OIDC access token |
| `X-Boot-Id` | optional | ULID-like value generated by the FE `BootTracer` at cold boot. Echoed into every backend structured log entry for that request so a login becomes one greppable story. Max length 64 chars; longer values rejected as malformed. Constant `BOOT_ID_HEADER` is shared via `@tagea/session`. |
| `X-Preferred-Institution-Id` | optional | Soft hint derived from the user's last successful landing. Backend MAY use it to disambiguate equivalent landings but always validates against authoritative state. Constant `PREFERRED_INSTITUTION_HEADER`. |

**Response 200:** `Session` (full shape — see Section 2 below).

**Error codes:**

- `401` — token missing / invalid
- `403` — email not verified, pending approval, suspended — the snapshot is still returned with `landing.mode = 'blocked'` and a `blockedReason`; the FE renders the right error page based on it. 403 is reserved for genuine authorisation failures (e.g. token revoked).

**Idempotency:** Pure read. Re-running yields the same snapshot for the same `(principal, tenant)` pair within the lifetime of the underlying assignments.

## 2. The `Session` DTO

Shape lives in `packages/session/src/lib/session.dto.ts`. Reproduced here for the wire-format record; if this section and the source disagree, **the source wins** and this file gets updated in the same PR.

```ts
export interface Session {
  bootId: string;                    // echo of X-Boot-Id (empty string when absent)
  identity: SessionIdentity;
  tenant: SessionTenant;
  elevation: SessionElevation;
  features: SessionFeatures;
  permissions: SessionPermissions;
  landing: Landing;
  personaLabel: PersonaLabel;
  preferences: SessionEmployeePreferences | null; // null for clients
}

export interface SessionIdentity {
  authUserId: string;
  principalId: string;
  type: 'employee' | 'client';
  email: string;
  firstName: string;
  lastName: string;
  status: 'active' | 'pending_approval' | 'suspended' | 'deleted';
  avatarUrl: string | null;
  phoneNumber: string | null;
  roleName: string | null;
  personnelNumber: string | null;     // employee-only, null otherwise
  accessProofOfSalary: boolean;       // gehaltsnachweise feature gate
  emailVisible: boolean;              // personenverzeichnis opt-in
  phoneVisible: boolean;              // personenverzeichnis opt-in
  notificationPreferencesSeen: boolean; // welcome-dialog one-shot flag
}

export interface SessionTenant {
  id: string;
  name: string;
  available: { id: string; name: string }[];
  pushBrandId: string | null;         // replaces GET /tenants/current/push-brand
  theme: SessionTenantTheme;          // replaces GET /tenants/current/theme
}

export interface SessionTenantTheme {
  primaryColor: string;
  primaryColorDark: string;
  lightBackgroundColor: string;
}

export interface SessionElevation {
  isSuperAdmin: boolean;
  isTenantAdmin: boolean;
  isSchulungAdmin: boolean;
}

export type SessionFeatureFlag = { enabled: boolean; [extra: string]: unknown };
export type SessionFeatures = Record<string, SessionFeatureFlag | undefined>;

export interface SessionPermissions {
  tenant: string[];
  institutions: Record<string, InstitutionMembership>;
  teamspaces: Record<string, TeamspaceMembership>;
  client: string[] | null;            // null for employees
}

export interface InstitutionMembership {
  institutionId: string;
  name: string;
  role: { id: string; name: string };
  counselingEnabled: boolean;
  isActive: boolean;
  lastUsedAt: string | null;
  permissions: string[];              // effective: tenant ∪ inst − denied
  features: Record<string, boolean>;  // effective: tenant ∧ inst
}

export interface TeamspaceMembership {
  teamspaceId: string;
  name: string;
  role: { id: string; name: string };
  scopedInstitutionIds: string[];
  permissions: string[];
}

export interface Landing {
  mode: 'teamspace' | 'einrichtung' | 'client_portal' | 'blocked';
  scope:
    | { kind: 'teamspace' }
    | { kind: 'institution'; id: string }
    | { kind: 'client_portal' }
    | { kind: 'none' };
  rationale: string[];
  blockedReason?: 'no_tenant' | 'no_assignments' | 'pending_approval' | 'email_not_verified';
}

export type PersonaLabel =
  | 'blocked'
  | 'pending_approval'
  | 'client_portal_user'
  | 'super_admin'
  | 'tenant_admin_no_institution'
  | 'multi_institution_employee'
  | 'single_institution_employee'
  | 'teamspace_only_employee';

export type SessionEmployeePreferences = Record<string, unknown>;
```

> **Flutter port note:** Map the Dart classes through `json_serializable`. The discriminated union `LandingScope` requires a manual `fromJson` factory; the rest is plain JSON.

## 3. Switch endpoints (M2)

### `POST /auth/current-tenant`

**Auth:** Bearer JWT. **Request body:**

```ts
{ tenantId: string }   // must be in user's available tenants
```

**Response 200:** `{ user, currentTenant, … }` (legacy DTO — FE ignores body and reloads the page).

**FE follow-up:** `SessionSwitcher.setTenant()` triggers `window.location.href = '/'` so `SessionBootstrap.hydrate()` runs cleanly under the new tenant context. Interceptor caches, Matrix client, push subscription, and route tree all rebuild from scratch.

**Errors:** `401` (tenant not in user's `AuthUserTenant` mappings), `404` (no principal in target tenant).

---

### `POST /auth/current-institution`

**Auth:** Bearer JWT. **Request body:**

```ts
{ institutionId: string | null }    // null clears the active institution
```

**Response 200:**

```ts
{
  success: boolean;
  institutionId: string | null;
  institutionName: string | null;
}
```

**FE follow-up (soft switch — no reload):**
1. `InstitutionContext.setInstitutionId(id)` — sticky-signal update
2. `DashboardDataService.clear()` + `ClientsDataService.clear()` + `CalendarDataService.clear()` — cache invalidation
3. `SessionBootstrap.hydrate()` — re-pulls the snapshot so the per-institution permission slice reflects the new institution

**Errors:** `400` (institution not assigned, not active, or counseling mode disabled).

---

### `PUT /session/preferences/last-visited-institution`

**Auth:** Bearer JWT. **Request body:**

```ts
{ institutionId: string }   // UUID v4, length 36
```

**Response:** `204 No Content`

Persists `auth_user_tenant.last_visited_institution_id` for `(authProviderUserId, tenantId)`. Read at session-assemble time by `LandingResolver.pickPreferredInstitution()`.

**Caller pattern (FE):** `SessionPreferencesService.recordVisitedInstitution(id)` — fire-and-forget on every `/einrichtung/:id/*` navigation, called inside `sessionInstitutionUrlGuard`. Does NOT block navigation.

**Errors:** `400` if no authenticated principal, no tenant context, or non-UUID payload.

## 4. Slim helper endpoints (M5+ Cluster 1)

### `GET /employees/me/matrix`

**Auth:** Bearer JWT. **Response 200:**

```ts
{ mxid: string | null }
```

Slim endpoint introduced in `874894561` (M5+ Cluster 1.3) so the chat-init code path doesn't have to fetch the full `GET /employees/me` (~750 B) just to read one field (~30 B response). Caller: `EmployeeSelfService.getOwnMatrixId()`.

### `PATCH /employees/me/matrix`

**Auth:** Bearer JWT. **Request body:** `UpdateMatrixIdDto = { mxid: string | null }`. Sets / clears the Matrix ID for the authenticated employee.

### `PATCH /employees/me/preferences`

**Auth:** Bearer JWT. **Request body:** `Record<string, unknown>` (jsonb merge). Read-back of preferences happens via `GET /session` — there is no GET counterpart.

> The FE keeps a defensive HTTP fallback (`getPersonalPreferences()` in `EmployeeSelfService`). Whether to retire that fallback is a follow-up — see spec §14.5.

## 5. Retained legacy endpoints (with documented reasons)

### `GET /auth/me/tenants`

**Retained because:** chat-frontend calls this as a pre-tenant-context endpoint. Cannot be deleted without a separate chat-frontend migration. Doc-comment in the controller marks it as do-not-extend.

### `POST /auth/me/change-password`

**Retained because:** legitimate action endpoint — Keycloak-bypass for in-app password change. Not consolidated into `/session` because it's a write, not a read.

## 7. Migration writes (M2 schema additions)

### Column: `auth_user_tenant.last_visited_institution_id`

```sql
ALTER TABLE meta.auth_user_tenant
  ADD COLUMN last_visited_institution_id UUID NULL;
```

- Nullable — null means "no preference yet, use fallback".
- No FK constraint to `institutions` because institutions live in tenant-DB shards, not in `meta`. Validated at read time by `LandingResolver.pickPreferredInstitution()` (membership must still exist and be active).
- Written by `SessionPreferencesService.setLastVisitedInstitution()`.

Migration: `apps/tagea-backend/src/migrations/meta/…AddLastVisitedInstitutionId.ts` (see Phase A history).

## 9. Verification note — known `verify-contracts.js` false positives

`specs/_scripts/verify-contracts.js auth-session` reports DRIFT on 9 identifiers because the script only searches `apps/tagea-frontend/src/`. The auth-session refactor spans three roots:

- **shared DTO** — `packages/session/src/lib/session.dto.ts` (`Session*` interfaces, `TeamspaceMembership`, `InstitutionMembership`, …)
- **backend** — `apps/tagea-backend/src/auth/session/` (`pickPreferredInstitution`, `setLastVisitedInstitution`, `blockedReason` literal)
- **frontend** — `apps/tagea-frontend/src/app/auth-session/`

Manual verification done 2026-05-12:

```bash
# Shared DTO types exist
grep -l 'SessionElevation\|SessionEmployeePreferences\|SessionPermissions\|SessionTenant\|TeamspaceMembership' \
  packages/session/src/lib/session.dto.ts apps/tagea-frontend/src/

# Backend symbols exist
grep -l 'pickPreferredInstitution\|setLastVisitedInstitution\|blockedReason' \
  apps/tagea-backend/src/auth/session/
```

The script's drift list for this spec is informational only; treat it as expected noise until the verify tool learns about `packages/` and `apps/tagea-backend/`.
